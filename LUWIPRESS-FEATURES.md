# LuwiPress — Complete Feature Overview

**Version:** 3.5.6 · **License:** GPLv2+ · **Target:** WooCommerce stores · **Tested with:** WordPress 7.0

LuwiPress is a standalone, AI-powered automation plugin for WordPress/WooCommerce. It generates content, optimizes SEO, translates products, and automates store management — integrating seamlessly with existing plugins (Rank Math, WPML, Elementor, etc.) without replacing them.

Shipped as a lean **~700 KB core** plus three optional companion plugins (WebMCP, Marketplace Sync, Agentic) — install only what your store needs.

- **Agentic Commerce — Google UCP + AP2 (3.6.0)** — LuwiPress now speaks the two open standards that let customers buy directly inside AI assistants. **Universal Commerce Protocol (UCP)** is Google's agentic-checkout standard for AI Mode in Search and Gemini; **Agent Payments Protocol (AP2)** is the payment-authorization layer beneath it (backed by Mastercard, PayPal, Adyen and 60+ others). A new **`LuwiPress → Commerce`** hub brings three capabilities. **(1) UCP feed readiness** — flag products eligible for Google's UCP "Buy" button (`native_commerce`), attach legal `consumer_notice` warnings for regulated items, map your feed id to your checkout id, and configure the store-level return policy + support info Merchant Center requires. Everything ships **default-OFF and sandbox-first** — nothing changes on your store until you turn it on. 15 new WebMCP tools expose the whole surface to AI agents (write-classified so the autonomous loop never places an order on its own). - **Vendor↔product bidirectional render robustness + canonical meta format (3.5.3)** — closes Tapadum FR-016 + a Knowledge Graph edge bug surfaced during live verification. **Root cause** was a latent format split: `_lwp_vendor_ids` product meta got written in two incompatible JSON shapes depending on which code path wrote it — the canonical admin save handler emitted `["123"]` (quoted strings) while third-party seed scripts + raw MCP `meta_set` calls produced `[123]` (bare integers). The vendor profile template's `LIKE` query was built around the quoted shape, so vendors seeded via direct DB writes showed correct `product_count` in the Knowledge Graph but rendered empty "work grids" on their profile pages (a Tapadum luthier with 4 attributed products had 0 cards on their page even though the KG counted them). **Three-part fix:** (1) the plugin now `register_post_meta`s `_lwp_vendor_ids` on the `product` post type with a `normalize_product_vendor_ids` sanitize_callback that accepts any incoming shape (JSON, comma-list, array, single integer) and validates each ID against an actual published vendor post before emitting the canonical `["123","456"]` form — every write through `update_post_meta` / REST / MCP / wp-cli / third-party hooks is normalized at the meta API boundary; (2) a one-time idempotent migration runs on `init` priority 20 (skipping AJAX/cron/REST sub-requests), walks every product carrying the meta, and re-saves through the sanitize callback so pre-existing integer-format data normalizes automatically on next page load after upgrade — flag stored in `luwipress_vendor_ids_normalized` option so subsequent loads no-op; (3) companion theme (luwipress-gold 1.10.3) `single-lwp_vendor.php` + `lwp-master-grid` widget switched from `LIKE` to `REGEXP` with JSON-array-element-aware boundaries `(\[|,)[[:space:]]*"?<id>"?[[:space:]]*(,|\])` so both formats match AND substring false-positives (`36633` against `366331`) are impossible. **Companion KG edge fix:** when `/knowledge-graph?sections=vendors` was called without `products` in the same request, the bulk loader returned correct vendor `product_count` per node but ZERO `made_by` edges — so the vendor view rendered isolated nodes instead of a supply graph. New `load_vendor_product_counts_with_edges` helper emits both counts AND edges from a single bulk query. **Theme companion 1.10.2 → 1.10.3 — `lwp-product-grid` widget shipped:** Pro-free curated product grid widget with native Elementor controls (limit / columns / orderby / category filter / featured-only mode / manual ID list) — drops in cleanly where a legacy `[products limit=8 columns=4]` shortcode used to live, preserves the `tap-products` (or any) wrapper class for legacy CSS hooks, renders with the same `lwp-pcard` chrome used on archive + related grids for visual consistency.

- **WPML translation-term slug-collision fix (WebMCP 1.0.19)** — the `taxonomy_update_term` MCP tool was rejecting every WPML translation-term update with "slug already in use" because `wp_update_term()` always re-runs `wp_unique_term_slug()` and that function isn't WPML-aware — sibling-language terms in different languages (e.g. `darbuka-argilla` IT vs `darbuka-arcilla` ES vs `darbuka-clay` EN) were falsely flagged as duplicates. Two-layer fix: description-only updates bypass `wp_update_term` entirely via a direct `$wpdb->update` on `wp_term_taxonomy` plus `clean_term_cache` + `edited_term` / `edited_{taxonomy}` action emission (response carries `method: "direct_description_write"`); full updates that touch name/slug/parent call `$sitepress->switch_lang($term_lang)` before `wp_update_term` so the slug-uniqueness check scopes to siblings in the term's own language (response carries `method: "wp_update_term"`). Unblocks bulk SEO-description sweeps across translated taxonomy terms on multilingual stores — a single MCP loop can now update hundreds of translated category descriptions without "slug already in use" rejections.

- **Slug Resolver redirect-loop fix + pre-swap audit gate (3.1.59)** — closes a P0 issue in the 3.1.58 nested-leaf rescue: URLs that were *already* `product-category/...` archives could match the nested-leaf map and silently 301 back to themselves, producing 5-hop redirect loops on menu visitors. ### 15. Agentic Commerce — Google UCP + AP2 (since 3.6.0)

Lets customers buy your products directly inside AI assistants — Google AI Mode (Search) and Gemini — through two open standards. You stay the **Merchant of Record**: the customer relationship, the order, and the data remain yours. All of it is **default-OFF and sandbox-first**, so nothing changes on your store until you switch it on from `LuwiPress → Commerce`.

**Universal Commerce Protocol (UCP) — feed readiness**

- **`native_commerce` eligibility** — flag which products qualify for Google's UCP "Buy" button, per-product or store-wide-by-default
- **`consumer_notice`** — attach mandatory legal warnings (e.g. California Prop 65) for regulated items
- **`merchant_item_id` mapping** — bridge your feed id to your checkout id when they differ
- **Merchant-of-Record settings** — return policy (cost + window + link) and customer-support contact, which UCP shows on the agentic checkout screen
- **Eligibility validator + coverage report** — per product, see exactly what blocks eligibility (no price, out of stock, missing image, weak identifiers, no return policy); store-wide, see how many products are ready
- **Supplemental feed** (JSON / CSV / XML) — overlays your existing Merchant Center feed with just the UCP signals, so your primary product data is never touched
- **Endpoints**: `GET|POST /ucp/settings`, `GET /ucp/eligibility`, `GET|POST /ucp/product/{id}`, `GET /ucp/feed`

**UCP Native Checkout**

- **Three UCP endpoints** — session create / update / complete — so an agent builds a cart, resolves real tax + shipping, and places the order in-conversation
- **Backed by WooCommerce** — each session is a WooCommerce `checkout-draft` order, so tax, coupons, and shipping rates all come from WooCommerce's own engine; totals are always recomputed server-side (a client-supplied total is never trusted)
- **Sandbox-first** — sandbox mode simulates completion with no payable order; live mode hands off a pending order for the payment processor to capture
- **Idempotent** — completing a session twice returns the same order; an optional idempotency key reuses an open session on retry
- **Endpoints**: `POST /ucp/checkout/session`, `GET|POST /ucp/checkout/session/{id}`, `POST /ucp/checkout/session/{id}/complete`

**Agent Payments Protocol (AP2) — mandate audit trail**

- **Mandate verification** — when an agent presents a signed Cart Mandate, LuwiPress checks its structure, expiry, and issuer allowlist
- **Amount match** — "what you see is what you pay for": the mandate's committed total must equal the order total, an independent merchant-side guard
- **Non-repudiable audit trail** — the Intent → Cart mandate chain and the verification verdict are stored on the order

## 🔐 Security & Performance

### Authentication (5 methods)
- **JWT Tokens** — RFC 7519 compliant, cryptographic secrets
- **Bearer Token** — LuwiPress API token (`lp_xxx`)
- **Session Cookie** — WP admin users
- **HMAC Signatures** — webhook signing
- **IP Whitelist** — restrict source IPs

### Security Features
- SQL injection prevention (parameterized queries)
- XSS protection (esc_html, wp_kses)
- CSRF nonce validation on admin AJAX
- Rate limiting per IP (1000 req/hour default)
- Token revocation by JTI
- Race condition guards (transient locks)
- MIME validation on uploads (10MB max)
- **Cache-layer isolation for authenticated REST responses** — every `luwipress/v1/*` response stamps `Cache-Control: no-store, private` plus a matching LiteSpeed-specific header so upstream page caches (LiteSpeed, Varnish, NGINX FastCGI, Cloudflare cache-everything) cannot replay an admin's body to a later unauthenticated visitor. Sensitive endpoints — CRM revenue, knowledge graph, settings, logs — are never served from a shared cache. Recommended belt-and-braces: also add `wp-json/luwipress` to your full-page cache plugin's "Do Not Cache URIs" list.

### Bot & Compliance Modules (3.1.60 / 3.2.0 / 3.2.1)
All three ship **DEFAULT-OFF** so existing installs aren't disrupted; enable per-module from the admin.

- **Bot Account Cleaner** (3.1.60+) — score-based detection (0–100) and safe bulk deletion of fake user accounts. Signals: disposable / temp-mail domain (200+ blocklist), `+` alias on Gmail/Outlook/Yahoo, high-entropy local part, numeric-tail patterns, 0 WC orders + 0 comments + never-logged-in + stale registration age, missing real name, burst registration windows. Hard-coded safety invariants protect admin/editor/shop_manager/author/contributor roles and any WC customer with at least one order. Dry-run by default; server re-scores each user before deletion even if the caller tries to bypass.
- **Cookie Consent** (3.2.0+) — GDPR/ePrivacy banner + preferences modal + consent log + AI policy generator. Three modes (info / opt-in / opt-out) × four categories (necessary / analytics / marketing / personalization). Frontend is vanilla JS (no jQuery), light/dark/auto theme, multilingual. Script blocking via `<script type="text/plain" data-luwipress-consent="analytics">…</script>` pattern — runtime rewrites after consent. IP stored as SHA-256-salted hash only (raw IP never persisted — Article 7(1) proof without GDPR risk). Public REST endpoint throttled to 5 writes/min per IP. AI policy generator passes the Plugin Detector's detected analytics/marketing/Meta tags as context so the generated paragraph names the real third parties on the site.
- **Bot Shield** (3.2.0+) — six-layer front-edge bot filter at `init` priority 1: UA blocklist (22 known scrapers by default — AhrefsBot, SemrushBot, MJ12bot, sqlmap, WPScan, etc.), per-IP rate limiter on sensitive paths (`/wp-login.php`, `/xmlrpc.php`, `/wp-json/wp/v2/users`), REST user-enumeration block (refuses `/wp-json/wp/v2/users` + `?author=N` for non-logged-in callers), XML-RPC kill switch + pingback-multicall removal, honeypot trap (auto-bans IPs probing `/wp-admin/install.php`, `/.env`, etc.), Googlebot/Bingbot reverse-DNS verification (defeats UA spoofing). Hard guards: logged-in admins + RFC1918 private IP space never auto-blocked. Dry-run test endpoint previews any (IP, UA, path) verdict without firing the shield — tune thresholds without lockout risk.
- **Wordfence-aware delegation** (3.2.1+) — when Wordfence is active, Bot Shield automatically delegates three layers to it (UA blocklist, rate limit, honeypot) to avoid double-blocking. LuwiPress keeps the layers Wordfence Free does not cover: cookie consent, REST user enumeration block, XML-RPC kill, allowlist, manual blocks, dry-run testing. Admin banner shows the delegation state; MCP `bot_shield_stats` includes `friendly_security` + `delegated_layers` fields.
- **Bot Shield: Comment Review** (3.3.0+) — independent toggleable 7th layer that filters comment submissions through a multi-signal scorer at `preprocess_comment` priority 1. Signals (each contributes a weight): author URL filled, digit-heavy or keyboard-mash author name, body link density above a configurable max (default >2 links), URL-only / URL-dominant body, spam-token hits from a configurable list (viagra / casino / crypto signal / replica watch / etc.), duplicate body within 10-min per IP, IP already on Bot Shield blocklist (auto-spam), random-shape email local-part. Threshold default 40. Four operator-selectable modes — **off**, **moderate** (hold for manual review, default — non-destructive), **spam** (route to spam queue), **reject** (silent 403 + optional auto-block of the IP). Logged-in users + allowlist always bypass. Last 100 caught events ring-buffered (`luwipress_bot_shield_comment_log` option) with score + matched signals + 240-char snippet for operator review. Admin panel on Bot Defense → Shield tab includes settings, dry-run test (paste a sample comment → verdict + signals), and the recent-events table. Third-party plugins can append custom signals via `luwipress_bot_shield_comment_extra_signals` filter.
