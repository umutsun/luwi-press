## P0 — Production security blockers

### 1. LiteSpeed caches authenticated REST responses by URL → cross-user auth bypass

**Severity:** P0 (actual customer data leaked without authentication)

**Reproduction:**
```bash
# Step 1 — authenticated request
curl -s "https://tapadum.com/wp-json/luwipress/v1/crm/overview?cb=1234" \
  -H "X-LuwiPress-Token: $TOKEN" > /dev/null

# Step 2 — same URL, no auth, different IP → returns the cached body
curl -sI "https://tapadum.com/wp-json/luwipress/v1/crm/overview?cb=1234"
# X-LiteSpeed-Cache: hit
curl -s  "https://tapadum.com/wp-json/luwipress/v1/crm/overview?cb=1234"
# → full 200 response including {"total_customers":6364,
#    "revenue_summary":{"lifetime":{"orders":248,"revenue":107874.51}},...}
```

**Endpoints confirmed vulnerable to cache-replay:**
- `/crm/overview` — customer counts, 30-day + lifetime revenue, segment distribution, lifecycle events
- `/crm/settings` — business thresholds (VIP spend, loyalty order count, recency windows)
- `/knowledge-graph` — full store intelligence graph (128 products, 57 categories, customer segments, opportunities)
- `/aeo/coverage` — product permalinks and names across all languages
- `/logs` — tool call history, admin IP addresses, user agents, execution times
- `/translation/missing` — 400 responses are cached too (body leaks params schema)

**Endpoints NOT affected (POST-only or correct headers):**
- `/crm/refresh-segments` (POST — LS doesn't cache POST)
- `/status`, `/health` (public already)
- 401 responses from endpoints not yet primed (`Cache-Control: no-cache, must-revalidate, no-store, private` sent correctly)

**Root cause:** Successful REST responses (200) are served without `Cache-Control: no-store` or `X-LiteSpeed-Cache-Control: no-cache` headers. LiteSpeed treats the URL as publicly cacheable. The first authenticated request "primes" the cache; subsequent anonymous requests to the same URL get `X-LiteSpeed-Cache: hit` and receive the full body.

**Response header diff:**
```
# Correct behaviour (on 401 responses):
X-LiteSpeed-Cache-Control: no-cache
Cache-Control: no-cache, must-revalidate, max-age=0, no-store, private

# Broken behaviour (on 200 responses):
Vary: Accept-Encoding
X-Litespeed-Cache: miss       # first request
X-LiteSpeed-Cache: hit        # subsequent requests — cross-user leak
# (no Cache-Control on the response)
```

**Suggested fix (pick one or layer them):**
1. **Centralised header filter** — in `LuwiPress_Permission::check_token` (or wherever auth decides), add a `rest_post_dispatch` filter that injects `Cache-Control: no-store, private` for every response whose endpoint required token/admin auth. LuwiPress already has `LuwiPress_Security` — adding a REST header filter there is 10 lines.
2. **Per-route cache-kill** — call `nocache_headers()` inside every auth-required handler at the top of the callback.
3. **LiteSpeed config (defence in depth)** — add `wp-json/luwipress` to the "Do Not Cache URIs" rule in LiteSpeed settings. But this should NOT be the only fix: fresh installs on other hosts inherit the same hole.

**Why this is P0, not P1:** the leaked data includes lifetime revenue ($107,874), customer count (6,364), 9 pending lifecycle events (with customer names + emails), and CRM threshold configuration. Any attacker who sees the URL of an admin's network trace (screen share, support ticket, referrer leak) can replay it from any IP.

## P1 — Functional bugs

### 2. `/product/enrich-batch/status` returns empty totals after batch completes

**Severity:** P2 (UX/observability — not a data-loss bug)

**Reproduction:**
```bash
# Queue a batch
curl -sX POST https://tapadum.com/wp-json/luwipress/v1/product/enrich-batch \
  -H "X-LuwiPress-Token: $TOKEN" -H "Content-Type: application/json" \
  -d '{"product_ids":[31422]}'
# → {"success":true,"batch_id":"batch_1c811aa5-...","queued":1,"errors":[]}

# Wait for completion (via /logs: "Batch enrichment completed locally: 1 products")

# Poll status
curl -s "https://tapadum.com/wp-json/luwipress/v1/product/enrich-batch/status?batch_id=batch_1c811aa5-..." \
  -H "X-LuwiPress-Token: $TOKEN"
# → {"batch_id":"batch_...","total":0,"completed":0,"progress":0,
#    "statuses":{"queued":0,"processing":0,"completed":0,"failed":0},"products":[]}
```

**Root cause (inspected [class-luwipress-ai-content.php:674-712](../luwipress/includes/class-luwipress-ai-content.php#L674-L712)):** The status handler joins `_luwipress_enrich_batch_id` and `_luwipress_enrich_status` postmeta. If either meta is cleaned up on completion, the row vanishes and `total` becomes 0. The `progress` calculation returns 0 when `total == 0` (not 100).

**Expected behaviour:** Completed batches should return `total=1, completed=1, progress=100` for at least 24h post-completion so operators can confirm what ran.

**Suggested fix:** Persist a batch-level record (e.g. in a separate transient or custom table) so historical status survives postmeta cleanup; OR stop cleaning the postmeta after success and instead age it out after 7d.

## P2 — Function bugs / UX issues

### 3. `/token-recent?limit=N` query param ignored

**Severity:** P2 (pagination missing on admin table)

Code at [class-luwipress-api.php:811](../luwipress/includes/class-luwipress-api.php#L811) hardcodes `get_recent_calls(20)`. Any `?limit=N` value is silently dropped. Admin dashboards that ask for e.g. limit=5 will get 20 rows and have to client-side slice.

**Fix:** Read `$request->get_param('limit')` with a sane default + max cap (`min(100, max(1, $limit ?: 20))`).

### 4. WebMCP accepts `jsonrpc: "1.0"` requests

**Severity:** P2 (spec compliance)

JSON-RPC 2.0 spec says `jsonrpc` MUST be exactly `"2.0"`. Our server currently processes requests with `"1.0"` and returns a valid `tools/list` response. Per the MCP spec (and general JSON-RPC etiquette) this should be rejected with `-32600` Invalid Request.

**Fix:** Reject any request where `$body['jsonrpc'] !== '2.0'` early in the dispatch.

### 5. AEO coverage across languages double-counts translations

**Severity:** P2 (dashboard consistency)

`/aeo/coverage` reports `total_products: 512, with_faq: 109, faq_coverage: 21.3%`.
`/knowledge-graph` reports `total_products: 128, aeo_coverage.faq: 47.7%`.

Both are technically correct (AEO counts language variants as separate posts; KG uses primary language), but the 2x discrepancy is confusing on first read. The Action Queue rubric uses the 47.7% figure, so KPIs are consistent there — but a store owner cross-checking the two pages will think one is broken.

**Fix options:**
- Have `/aeo/coverage` expose both `primary_language_total` and `all_languages_total`.
- OR align `/aeo/coverage` to primary-language only (matches KG).

### 6. Customer Chat handpan query returns handpan accessories, not handpans

**Severity:** P2 (RAG relevance)

First message "do you have darbukas?" → returns 2 actual darbukas. **PASS**.
Second message "What about handpans?" → returns 3 **handpan cases/bags**, not handpans (Tapadum does sell handpans — the Handpan Tapadum Pro line is in the catalog).

Indicates the BM25 + KG retrieval is leaning on "handpan" as a term match and surfacing whatever product co-occurs with it (cases named "Handpan Case", "Handpan Safety Bag"). Handpan Tapadum Pro products *are* in the uncovered AEO list, so they exist.

**Fix:** Either tune BM25 to prefer category-level matches (handpan category = 9-note, 12-note handpan scales), or filter out accessory post_types/categories when a customer asks about the instrument itself.

## P3 — Polish / cosmetic

### 7. `search_products` (WebMCP) returns mangled UTF-8

**Severity:** P3 (LLM consumption quality)

```
"description": "Darbuka de barro - ... Descubre el coraz�n r�tmico
                de la m�sica de Oriente Medio y el norte de �frica ..."
"price": "489.14&euro;"
```

Spanish accented characters appear as `�` (replacement char). Price carries HTML entity `&euro;` instead of `€` or `EUR`. Both indicate the tool runs `wc_get_product()->get_description()` (which applies WP's `the_content` filter leaving entities) and the raw JSON doesn't specify UTF-8 charset correctly for the transport layer.

**Fix:** `html_entity_decode( ..., ENT_QUOTES, 'UTF-8' )` on user-facing strings before packaging for MCP output. Price formatting can be normalised to `wc_price` with stripped tags.

### 8. WebMCP exposes no `woo_list_products` / product CRUD

**Severity:** P3 (tool surface inconsistency — not a bug if intentional)

WebMCP has `woo_list_orders`, `woo_get_order`, `woo_update_order_status`, `woo_list_coupons`, `woo_create_coupon`, `woo_get_shipping_zones`, `woo_get_tax_rates` — but no `woo_list_products`. Product listing is done via `content_get_posts` (WP post_type=product) and `search_products` (BM25). This is internally consistent, but an LLM reading the tools catalog alphabetically will wonder why products are the odd one out. Either document the convention in the `woo_*` tool descriptions, or add a thin `woo_list_products` alias.

## Priority fix list (recommended order)

1. **P0 — Kill the LiteSpeed REST cache leak.** Add `Cache-Control: no-store, private` header to every authenticated REST response. Smallest intervention: `add_filter('rest_post_dispatch', function($resp, $server, $req){ if (strpos($req->get_route(),'/luwipress/v1/')===0) $resp->header('Cache-Control','no-store, private'); return $resp; }, 10, 3);` in `LuwiPress_Security::__construct()`. Also document a "Do Not Cache URIs" entry for LiteSpeed admins as defence in depth.
2. **P2 — Batch status persistence.** Keep the `_luwipress_enrich_batch_id` + `_luwipress_enrich_status` postmeta for at least 24h post-completion so the status endpoint can show `total/completed/progress`.
3. **P2 — JSON-RPC version guard.** Reject `jsonrpc !== "2.0"` with `-32600` in `LuwiPress_WebMCP` dispatcher.
4. **P2 — `/token-recent` limit param.** Honor `$request->get_param('limit')` with default 20, cap 100.
5. **P2 — Align AEO coverage numerators.** Expose primary-language + all-languages variants; match KG number in the admin Usage page.
6. **P2 — Chat RAG: handpan vs. accessory retrieval.** Tune BM25 stop/boost lists or add category type-filter (product != accessory).
7. **P3 — `search_products` UTF-8 / entity decode.** Apply `html_entity_decode` on description + `wc_price` on price before packaging.
8. **P3 — Optional `woo_list_products` alias** or document why it doesn't exist in `woo_*` tool descriptions.
